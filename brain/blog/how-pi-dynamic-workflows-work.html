01 — The substrate

First, what is Pi?

Pi is a coding agent — a terminal program that talks to an LLM, reads your files, runs your tools, and edits your code. Think of it as the engine. By itself it knows how to have a conversation and use tools; it knows nothing about your team's particular rituals.

That's the gap extensions fill. An extension is a small TypeScript package that Pi loads at startup and hands a remote control to. With that remote you can add slash-commands, register new tools the model can call, listen to lifecycle events, draw custom terminal UI, and stash data in the session. Everything in this article is one such extension — but to read it, you only need three nouns:

🧠

The main agent

Pi's primary conversation loop — the model you're chatting with in the terminal.

🔌

The extension

Trusted code Pi loads. It adds commands, tools, events & UI. Runs with full system access.

🤖

The subagents

Throwaway side-conversations the extension can spawn — each with its own prompt & transcript.

02 — The handshake

How an extension plugs in

Pi discovers an extension through a pi manifest in package.json. It points at one entry file:

package.json

{
  "pi": {
    "extensions": ["./src/extension/index.ts"]
  }
}

That file default-exports a factory function. Pi loads it through jiti (so TypeScript just works, no build step) and calls it once with an ExtensionAPI object — the remote control. Our entire extension's top level is four lines:

src/extension/index.ts

export default function dynamicWorkflowExtension(pi: ExtensionAPI): void {
  registerWorkflowsCommand(pi);   // the /workflows monitor
  registerWorkflowStatusline(pi); // the footer status cue
  registerUltracode(pi);          // the trigger + the Workflow tool
}

From that pi handle, an extension reaches for a handful of capabilities. These are the verbs that make the whole feature possible:

CapabilityWhat it does
pi.registerCommandAdds a slash-command like /workflows with a handler.
registerTool (via the tool module)Exposes a tool the model can call — this is how the agent launches a workflow.
pi.on(event, …)Subscribes to lifecycle events: input, before_agent_start, tool_call, session_start…
ctx.ui.*Draws UI: notify(), a custom editor, a status line, full custom TUI components.
pi.appendEntry / ctx.sessionManagerPersists & restores extension state inside the Pi session.
The boundary that matters. Extension code (src/extension/*, src/workflows/*) is trusted — it runs with Pi's full permissions. The workflow scripts users write are untrusted orchestration. Keeping those two worlds apart is the central design tension of the whole feature.

03 — The idea

The dynamic workflow, borrowed from Claude Code

A single agent thinking step-by-step is powerful but serial. Some problems — "audit this package across six quality dimensions, then adversarially verify every finding" — beg to be fanned out. Claude Code shipped a feature for exactly this, and this project reverse-engineers it onto Pi.

The shape: the main agent writes a tiny JavaScript orchestration script and hands it to a Workflow tool. The script doesn't touch your files itself — it just coordinates. Its only superpower is agent(): a call that spins up a fresh, isolated subagent to go do real work. The runtime runs that script in the background, schedules the subagents, records everything, and pings the main agent when it's done.

A workflow script in full

export const meta = {
  name: 'webfetch-quality-audit',
  description: 'Audit the package, then adversarially verify each finding',
  phases: [
    { title: 'Review', detail: 'One reviewer per audit dimension' },
    { title: 'Verify', detail: 'Adversarially verify each finding' },
  ],
}

// Each dimension is reviewed; the moment a review lands, its findings
// fan out to verifiers — no waiting for the other reviewers.
const results = await pipeline(
  DIMENSIONS,
  (dim) => agent(dim.prompt, {
    label: `review:${dim.key}`, phase: 'Review',
    agentType: dim.agentType, schema: FINDINGS_SCHEMA,
  }),
  (review, dim) => parallel(review.findings.map((f) => () =>
    agent(buildVerificationPrompt(f), {
      label: `verify:${dim.key}:${f.id}`, phase: 'Verify',
      agentType: 'general-purpose', schema: VERDICT_SCHEMA,
    }),
  )),
)

return results
This isn't a thought experiment. That exact script ran. The reference audit launched 34 subagents — 6 reviewers, 28 verifiers — burned ~1M tokens across 299 tool calls in 255 seconds, and returned a per-dimension verdict. Every number in this article comes from real on-disk artifacts.

04 — The vocabulary

Seven globals, and nothing else

A workflow script is plain JavaScript with top-level await. But it runs in a cage (more on that next) where the only things in scope are these:

GlobalRole
agent(prompt, opts)Spawn one subagent. Returns its final text — or, with a schema, a validated object.
parallel(thunks)Run many agent() calls at once. A barrier: waits for all.
pipeline(items, …stages)Flow each item through stages independently. No barrier between stages.
phase(title) / log(msg)Drop a progress marker / a run-level log line.
argsWhatever input the launch passed in.
budget{ total, spent(), remaining() } — a hard token ceiling for the run.

parallel vs pipeline — the distinction the whole feature hinges on

Both fan out. The difference is when the next stage is allowed to start. Naively chaining parallel calls creates a barrier at every boundary — the fastest item waits for the slowest before anyone moves on. pipeline deletes those barriers:

parallel-of-stages barrier

Areview⌛ waitverify
Breview⌛ waitverify
Creview…slowverify

A & B finish review fast — then idle until C's slow review lands. Wasted wall-clock.

pipeline no barrier

Areviewverify
Breviewverify
Creview…slowverify

A & B start verifying the instant their own review is done. C runs on its own clock.

Why thunks, not promises? parallel takes () => agent(…), never agent(…). A bare call would start the agent immediately, sneaking past the scheduler's concurrency cap. Wrapping it in a function lets the scheduler decide when to pull the trigger.

05 — The cage

Running untrusted JS safely

The script is user-authored. It must not get filesystem, shell, network, or MCP access — every side effect has to flow through a subagent, where Pi's normal permission policy applies. So the runtime evaluates the body inside a Node node:vm context that exposes only the seven globals. No process, no require:

proven by test/workflows/script/runtime.test.ts

// inside a workflow script:
typeof process   // "undefined"
typeof require   // "undefined"
agent, parallel, pipeline, phase, log, args, budget  // ✓ the only things in scope

The determinism rule

Three innocent-looking calls are banned: Date.now(), Math.random(), and argument-less new Date(). Not for security — for resumability (§7). The ban is enforced twice so neither a direct call nor an alias slips through:

Parse time — AST walk

Before anything runs, the parser walks the script's syntax tree and rejects any literal Date.now() / Math.random() / new Date().

Run time — poisoned globals

The VM swaps in Date & Math objects whose now/random throw. So even const m = Math; m.random() dies at runtime.

Honest caveat. An ADR in the repo states plainly that node:vm is not a complete security boundary — it's an execution kernel for the first implementation, not a finished sandbox story. The architecture keeps trusted and untrusted code separable so a stronger isolate can drop in later.

06 — The traffic cop

One scheduler, two hard caps

Every agent() call — whether bare, inside a parallel, or a pipeline stage — funnels through a single run-wide scheduler. It enforces two limits:

⚡

maxConcurrent

min(16, cpuCores − 2). On the reference 12-core machine, the run peaked at exactly 10 live agents.

🛑

maxTotalAgents = 1000

A lifetime ceiling per run. Cross it and agent() throws — the runaway-loop backstop.

Each subagent moves through a small state machine. Queued agents wait for a slot; running agents resolve to one of three terminal states (a failed/stopped one can be restarted back to the queue):

  queued ──agent_started──▶ running ──agent_succeeded──▶ done   ✓
    │                          │
    │                          ├──agent_failed────────▶ failed ──┐
    │                          └──agent_stopped───────▶ stopped ─┤
    └──agent_stopped──▶ stopped          (restart) ◀─────────────┘

07 — The memory

The journal, and resume-without-snapshots

A long run can crash, get killed, or get edited mid-flight. Re-running 34 agents from scratch is expensive. The trick: never snapshot the JavaScript VM. Instead, re-run the script from the top and skip any agent call whose answer is already on disk.

Every meaningful transition appends one line to an append-only journal.jsonl. The key insight is the stable key: a SHA-256 hash of the effective call — prompt, schema, label, phase, agent type, model, cwd, runtime version. It is deliberately not the random agent id.

journal.jsonl — a v2 stable key, plus a separate random agentId

{"type":"started","key":"v2:c244b274…be6aa1","agentId":"afebf68095c3a82a6"}
{"type":"result","key":"v2:2a3da59b…b83f812","agentId":"aaca1bc181ea38afe",
  "result":{"dimension":"Architecture & Module Boundaries","findings":[…]}}

Resume then becomes a replay:

Rebuild the cache

Scan the journal top-to-bottom; map each key to its latest result. Started-but-never-finished keys are incomplete and excluded.

Re-run the script

Execute from line one. JavaScript variables reconstruct themselves naturally because the code runs again.

Hit or spawn

At each agent(), recompute the key. Cached? Return instantly, no subagent. Missing or incomplete? Spawn a fresh one.

This is why determinism is non-negotiable. If the script computed a key from Date.now() or a random value, the key would change on replay, every cache lookup would miss, and resume would re-run the entire swarm. Pass time in through args instead.

08 — The whole loop

From ultracode to swarm and back

Now the pieces connect. Because a workflow can spawn a thousand agents and cost real money, it's gated behind an explicit opt-in: the user must type ultracode <goal>. Here's the full lifecycle, all driven by the events the extension subscribed to:

sequenceDiagram autonumber participant U as User participant E as Extension participant M as Main agent participant R as Workflow runtime participant S as Subagents U->>E: ultracode "audit webfetch" Note over E: input event →
mode = ON, persist session entry E-->>M: transform input to the bare goal Note over E,M: before_agent_start →
inject policy + system prompt M->>E: calls Workflow tool (script) Note over E: tool_call → allowed only
because mode is ON E->>R: launchWorkflow() · returns runId immediately R->>S: schedule agent() calls (capped) S-->>R: results → journal + manifest R-->>M: task-notification (triggerTurn) M-->>U: continues from the result

The gate, in code

Two event handlers do the gating. input flips the mode on and rewrites the message; tool_call blocks the Workflow tool whenever the mode is off:

src/extension/ultracode/register-ultracode.ts

// The Workflow tool is refused unless the user opted in:
export function handleUltracodeToolCall(event, mode) {
  if (event.toolName !== WORKFLOW_TOOL_NAME) return undefined;
  if (isUltracodeModeActive(mode)) return undefined;   // allow
  return {
    block: true,
    reason: "Workflow is disabled outside ultracode because it can spawn " +
      "many subagents and be expensive. Ask the user to type `ultracode " +
      "` to opt into multi-agent orchestration, then retry.",
  };
}

When the run finishes, the runtime fires a <task-notification> back into the conversation. For ultracode-launched runs it uses pi.sendMessage(…, { triggerTurn: true }) so the main agent is re-invoked with the result rather than just shown a new message — it picks up where the swarm left off.

09 — On disk

Where everything lives

A run leaves a clean, auditable trail. The manifest.json is the cheap read-model the UI loads; the journal and transcripts are the deep audit layer. Saved workflows live outside any run — they're just reusable orchestration.

# a Pi workflow root, walked up from cwd <pi-workflow-root>/ ├─ wf_901813da-ebe/ ← one run │ ├─ manifest.json read-model · the ONLY file /workflows needs │ ├─ script.js exact script this run executed │ ├─ journal.jsonl append-only resume/cache log │ ├─ output.json full terminal result │ └─ transcripts/ agent-<id>.jsonl + .meta.json ×34 │ └─ webfetch-quality-audit.js a project-local saved workflow (no run state)
The read-model rule. The /workflows list view must render from manifest.json alone — never parsing journals or transcripts. Loading a 132 KB journal and 3.9 MB of transcripts just to draw an overview row would be absurd. Cheap to read = snappy UI.

11 — Step back

The mental model, in one breath

Pi is the engine. An extension is trusted code that hands the agent new verbs. This particular extension's verb is Workflow — a sandboxed JS runtime where a script written by the agent fans out into a swarm of isolated subagents, all funneled through one capped scheduler, all journaled so the whole thing can resume without ever snapshotting a VM, and all watchable from a live /workflows monitor. The cost gate (ultracode), the determinism ban, the thunk requirement, the read-model rule — each is a small, deliberate constraint that makes the big, expensive capability safe to hand to a model.

🔌

Extensions = verbs

Commands, tools, events, UI — bolted onto the agent at load time.

🧬

Scripts orchestrate

They never do I/O. They only schedule subagents that do.

📓

Journal = resume

Replay the script, skip cached calls. Determinism makes it sound.

🛡️

Constraints = safety

Caps, gates & sandbox turn a swarm into something you can trust.

That's the whole machine.

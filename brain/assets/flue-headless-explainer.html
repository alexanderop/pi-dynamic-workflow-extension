Concept 1

Concept 2 · the catch

But an agent needs a computer to do things.

Agents run shell commands — ls, grep, cat, write files. The obvious way to give each agent a safe place to do that is to boot a real container or mini-VM per agent. That's fine for one. Now do it for a thousand.

Concept 3 · the trick

just-bash: skip the OS. Fake the computer in memory.

just-bash is a pretend bash shell + a pretend filesystem, written in TypeScript, living inside the same process as your app. No OS to boot. The "filesystem" is just a JavaScript object in RAM. So spawning an agent's computer is basically free.

The honest tradeoff ⚖️

Why it's great

Instant, tiny, scales to thousands, runs anywhere JS runs (even Cloudflare Workers). Network off by default.

The catch

It runs in your process — it is not a hard security wall like a real VM. For untrusted/risky code, Flue tells you to use a real sandbox instead.

Concept 4 · bringing it home

So... should your project use just-bash?

🙅

No — and here's the one-line reason.

Flue fakes a computer because nobody's there and it runs a thousand strangers' agents on a server. Your Pi extension is the opposite: it runs on the user's own machine, with the user right there. The agent already has a real shell and real files — the user is the computer. There's no per-agent container cost to dodge, so there's nothing for just-bash to save.

(And note: just-bash is a fake bash shell. Your node:vm runs the workflow orchestration script. Different layers — not a swap.)
